# 🤖 AI에게 일 잘 시키는 시스템 가이드

> 8년차 개발자가 Claude Code로 6개월간 책 300권 분량의 프로그램을 혼자 재구축한 경험에서 추출한 시스템.
> AI는 그냥 쓰면 50점, 시스템을 만들어주면 95점이 된다.

## 0. 매뉴얼 계층 구조: CLAUDE.md vs SKILL.md

Claude Code가 참조하는 매뉴얼은 **두 가지 레벨**로 동작한다. 모든 프로젝트에서 이 구조를 기본으로 세팅하라.

```
프로젝트 루트/
├── CLAUDE.md          ← 🔴 항상 자동 로드 (글로벌 규칙)
├── src/
│   ├── frontend/
│   ├── backend/
│   └── ...
└── .claude/
    └── skills/        ← 🟡 조건부 자동 로드 (전문 매뉴얼)
        ├── frontend/
        │   └── SKILL.md
        ├── backend-api/
        │   ├── SKILL.md
        │   └── references/
        ├── database/
        │   └── SKILL.md
        └── testing/
            └── SKILL.md
```

### CLAUDE.md — 항상 읽히는 글로벌 규칙

| 항목 | 설명 |
|------|------|
| **로드 시점** | 매 대화 시작 시 **무조건** 자동 로드 |
| **용도** | 프로젝트 전체에 적용되는 규칙 |
| **적합한 내용** | 코딩 스타일, 네이밍 컨벤션, 금지 패턴, 프로젝트 구조 설명, 커밋 규칙 |
| **주의** | 너무 길면 매번 토큰 낭비 → 핵심만 간결하게 |

### SKILL.md — 조건 맞을 때 자동으로 읽히는 전문 매뉴얼

| 항목 | 설명 |
|------|------|
| **로드 시점** | description의 트리거 조건에 매칭될 때 **자동** 로드 |
| **용도** | 특정 영역(프론트, 백엔드, DB 등) 전문 규칙 |
| **적합한 내용** | 해당 영역의 상세 패턴, 에러 처리 방식, 보안 규칙, 참조 문서 |
| **핵심** | **description을 공격적으로 잘 써야 트리거된다** |

### 작동 흐름

```
사용자: "API 엔드포인트 추가해줘"
         │
         ▼
  ① CLAUDE.md 자동 로드 (항상)
    → 프로젝트 전체 규칙 적용
         │
         ▼
  ② "API", "엔드포인트" 키워드 감지
    → backend-api/SKILL.md description 매칭
    → SKILL.md 자동 로드
         │
         ▼
  ③ 필요 시 references/ 상세 문서 추가 로드
    → auth.md, error-handling.md 등
         │
         ▼
  ④ 모든 규칙이 적용된 상태에서 작업 수행
```

### 새 프로젝트 시작 시 기본 세팅 체크리스트

- [ ] `CLAUDE.md` 작성 — 프로젝트 구조, 기술 스택, 글로벌 컨벤션
- [ ] 주요 영역별 `SKILL.md` 생성 — 프론트엔드, 백엔드, DB, 테스트 등
- [ ] 각 SKILL.md의 description에 트리거 키워드/의도/파일경로 조건 명시
- [ ] 300줄 이상의 상세 규칙은 `references/`로 분리
- [ ] 작업 기억용 문서 저장 폴더 지정 (계획서, 맥락노트, 체크리스트)

## 0-1. 실행 정책: 멈추지 말고 끝까지

### 모든 권한 허가 (ALL ALLOW)

- 프로젝트 관련 모든 작업에 대해 **전체 권한이 부여**된 상태다
- 파일 생성, 수정, 삭제, 패키지 설치, 빌드, 테스트 실행 등 **모든 행위가 사전 승인**되어 있다
- 작업 도중 권한 확인, 승인 요청, 진행 여부 질문을 **하지 마라**

### 중단 없는 연속 실행

- 작업이 시작되면 **완료될 때까지 멈추지 않는다**
- 중간에 "이렇게 진행해도 될까요?", "계속할까요?" 같은 질문 금지
- 멀티 스텝 작업은 순차적으로 끊김 없이 실행한다
- 모호한 부분이 있으면 **가장 합리적인 판단으로 직접 결정**하고 진행한다

### 완료 기준

- 최종 결과물이 **즉시 사용 가능한 상태**여야 완료다
- 빠진 단계가 없고, 지시의 의도(문자 그대로가 아닌 실제 목적)에 부합해야 한다
- 한 가지 방법이 실패하면 **대안을 시도**한 후에야 보고한다

## 0-2. 코드 수정 정책: 기존 코드를 함부로 건드리지 마라

### 기본 원칙: 기존 코드 보존

- **임의로 기존 코드를 수정하지 않는다** — 이것은 절대 규칙이다
- 새 기능 추가 시 기존 로직에 영향을 주지 않는 방식으로 구현한다
- "더 나은 방식"이라는 이유로 동작하는 기존 코드를 리팩토링하지 않는다

### 수정이 필요한 경우의 프로세스

```
1. 수정이 불가피하다고 판단되면
   → 어떤 파일의 어떤 부분을 왜 수정해야 하는지 먼저 보고

2. 사용자가 수정을 요청한 경우에도
   → 수정 대상 파일과 변경 범위를 먼저 나열
   → "다음 파일들이 수정됩니다: ○○, ○○" 형태로 확인

3. 확인 후 수정 실행
   → 변경 전/후를 명확히 기록
```

### 금지 행위

- ❌ 지시받지 않은 파일을 "개선" 목적으로 수정
- ❌ 작업 범위 밖의 코드 스타일 변경
- ❌ "이왕 하는 김에" 식의 추가 리팩토링
- ❌ 수정 범위를 알리지 않고 바로 변경

## 1. 자동 매뉴얼 시스템 — 스킬(Skill) 활용 (가장 중요)

### 문제
매뉴얼(스킬)을 아무리 잘 작성해도 AI가 스스로 읽지 않는다.

### Claude Code의 스킬(Skill)이란?

스킬은 Claude Code에 내장된 **업무 매뉴얼 시스템**이다. `SKILL.md` 파일에 작업 규칙과 가이드라인을 작성하면, Claude가 관련 작업 시 이를 참조한다.

#### 스킬 폴더 구조

```
skill-name/
├── SKILL.md              ← 핵심 매뉴얼 (필수)
│   ├── YAML frontmatter   (name, description)
│   └── Markdown 본문       (실제 지침)
└── 번들 리소스 (선택)
    ├── references/        ← 상세 문서 (필요 시 로드)
    ├── scripts/           ← 자동화 스크립트
    └── assets/            ← 템플릿, 아이콘 등
```

#### SKILL.md 작성 예시

```markdown
---
name: backend-api
description: "백엔드 API 개발 매뉴얼. 백엔드, 서버, API, 
  엔드포인트, REST, GraphQL 관련 작업 시 반드시 이 스킬을 참조하라.
  새 API 추가, 수정, 에러 처리, 인증 관련 작업이 모두 해당된다."
---

# 백엔드 API 개발 가이드

## 필수 규칙
- 모든 API는 에러 처리를 포함할 것
- 인증 미들웨어를 반드시 적용할 것
- ...

## 상세 문서
- 인증 관련: `references/auth.md` 참조
- DB 쿼리 패턴: `references/database.md` 참조
```

### 스킬이 자동으로 읽히게 만드는 핵심: Description 최적화

Claude는 스킬의 `description` 필드를 보고 해당 스킬을 읽을지 판단한다. **description이 부실하면 스킬을 무시한다.**

#### ❌ 나쁜 description
```yaml
description: "백엔드 개발 가이드"
```
→ 너무 짧고 모호해서 트리거 안 됨

#### ✅ 좋은 description
```yaml
description: "백엔드 API 개발 매뉴얼. 백엔드, 서버, API, 
  엔드포인트, REST, 라우터, 미들웨어, 인증, DB 쿼리 관련 
  작업 시 반드시 이 스킬을 참조하라. 새 API 생성, 기존 API 
  수정, 에러 처리 추가, 보안 검토가 모두 해당된다. 
  사용자가 명시적으로 요청하지 않더라도 해당 영역 파일을 
  수정할 때는 항상 참조할 것."
```
→ **키워드, 의도, 작업 범위를 모두 명시**하여 자동 트리거율 극대화

### 해결: 강제 매뉴얼 로딩 (Hook 시스템)

description만으로 부족할 때, **Hook(훅)**으로 강제 로딩한다.

#### ① Pre-hook (시작 전 알림 장치)
- 사용자의 지시 내용을 분석하여, 작업 시작 전에 관련 매뉴얼을 자동으로 로드
- 비유: "야, 이 업무는 매뉴얼 3번 참고해야 돼. 꼭 펴봐."

#### ② Post-hook (완료 후 체크 장치)
- 작업 완료 후 결과물을 점검하며 누락 사항을 상기
- "에러 처리는 추가했나요?", "보안 확인했나요?"

### 매뉴얼 자동 활성화 4가지 조건

| 조건 | 트리거 방법 | 예시 |
|------|------------|------|
| **키워드** | description에 관련 키워드 나열 | "백엔드", "서버", "API" → 백엔드 스킬 활성화 |
| **의도 파악** | description에 작업 유형 명시 | "새 기능 만들어줘" → 해당 영역 스킬 활성화 |
| **작업 위치** | description에 폴더/파일 경로 조건 명시 | `src/api/` 폴더 수정 시 → 백엔드 스킬 활성화 |
| **파일 내용** | description에 파일 패턴 조건 명시 | 파일 내 `@Controller` 패턴 → 백엔드 스킬 활성화 |

### 핵심 교훈: 3단계 점진적 로딩 (Progressive Disclosure)

1,500줄짜리 매뉴얼을 통째로 읽히면 토큰 낭비다. **3단계로 분리**한다:

```
1단계: 메타데이터 (항상 로드)
       → name + description (~100단어)
       → Claude가 스킬을 쓸지 판단

2단계: SKILL.md 본문 (트리거 시 로드)
       → 핵심 규칙과 가이드 (~500줄 이내)
       → 상세 문서 참조 포인터 포함

3단계: references/ 파일 (필요 시만 로드)
       → 상세 챕터별 문서 (용량 무제한)
       → 실제 필요한 챕터만 열람
```

#### 분리 구조 예시

```
backend-api/
├── SKILL.md                    ← 핵심 규칙 + 목차 (500줄 이내)
└── references/
    ├── auth.md                 ← 인증 관련 상세 가이드
    ├── database.md             ← DB 쿼리 패턴
    ├── error-handling.md       ← 에러 처리 규칙
    └── security-checklist.md   ← 보안 점검 체크리스트
```

- ❌ 1,500줄 통째 매뉴얼 → AI 토큰 낭비, 핵심 놓침
- ✅ 목차(SKILL.md) + 상세 챕터(references/) → 필요한 것만 로드
- **결과: AI 토큰 소비 40~60% 절감**

### 실전 팁: 스킬 작성 시 체크리스트

- [ ] description에 트리거 키워드를 충분히 넣었는가?
- [ ] description에 "명시적 요청 없어도 참조하라"는 문구가 있는가?
- [ ] SKILL.md 본문이 500줄을 넘지 않는가?
- [ ] 300줄 이상의 상세 내용은 references/로 분리했는가?
- [ ] references/ 파일에 목차(TOC)가 있는가?
- [ ] 언제 어떤 reference 파일을 읽어야 하는지 SKILL.md에 명시했는가?

## 2. 작업 기억 시스템

### 문제
AI는 자신감은 넘치지만 기억력이 금붕어다. 대화가 길어지면 원래 목표를 잊는다.

### 해결: 3개 문서로 외부 기억장치 구축

| 문서 | 역할 | 건축 비유 |
|------|------|-----------|
| **계획서** | 무엇을 만들 건지 처음부터 끝까지 | 설계도 |
| **맥락 노트** | 왜 이렇게 결정했는지, 관련 자료 위치 | 시방서 |
| **체크리스트** | 뭘 끝냈고 뭐가 남았는지 추적 | 공정표 |

### 작업 프로세스

```
1단계: AI에게 계획을 세우게 한다
       "이번 프로젝트 어떻게 진행할 건지 계획부터 세워봐"

2단계: 계획을 꼼꼼히 검토한다 (매우 중요!)
       → AI가 핵심을 오해하는 경우가 많다

3단계: 승인 후 즉시 문서로 저장
       "잠깐, 먼저 이 계획을 문서로 저장해"
       → 바로 실행하려는 AI를 멈춰야 한다

4단계: 새 대화에서 문서 읽고 작업 이어가기
       "저장된 문서 읽고 이어서 작업해"
```

### 핵심 팁

- **한 번에 전부 시키지 마라** → 한두 개 작업만 시키고 확인 후 다음으로
- **중간중간 체크리스트 업데이트** → "방금 한 거 체크하고 다음 할 일 정리해"
- 이렇게 하면 새 대화를 시작해도 AI가 길을 잃지 않는다

## 3. 자동 품질 검사 시스템

### 문제
AI가 "다 했습니다"라고 자신 있게 말하지만 실수가 수두룩한 경우가 많다.

### 해결: 공장 검수 라인 구축

#### ① 수정 기록 장치
- AI가 파일 수정 시 어떤 파일을 건드렸는지 자동 기록 (CCTV 역할)

#### ② 완료 후 검사 장치
- ❌ 작업 중간에 검사 → 아직 미완성 상태라 무의미 (요리 중간에 간 보는 격)
- ✅ **작업 완전 완료 후** 검사 실행
- 검사 내용:
  - 수정 내용 확인
  - 오류 자동 체크
  - 오류 적으면 → AI에게 즉시 수정 지시
  - 오류 많으면 → 전문 수리 담당(에이전트) 투입

#### ③ 셀프 체크 리마인더
- 작업 완료 시 자동 알림: "에러 처리 추가했나요?", "보안 위험은 없나요?"
- 강제가 아닌 상기 → AI가 스스로 보완하게 유도

## 4. 전문 에이전트 (AI 팀 구성)

### 개념
하나의 AI에게 모든 걸 시키지 말고, 역할별 전문 팀원으로 분리한다.

### 팀 구성 예시
- **품질 관리 팀** - 코드 품질 점검
- **테스트 팀** - 테스트 코드 작성 및 실행
- **기획 팀** - 설계 및 구조 검토
- **코드 리뷰 담당** - 다른 AI의 결과물을 검토 (자기 리뷰지만 효과적)

### 핵심: 구체적 보고서 요구

- ❌ "다 했습니다" (무슨 일을 했는지 모름)
- ✅ **"무엇을 발견했고, 무엇을 수정했고, 왜 그렇게 판단했는지 보고서로 정리해서 와라"**

## 전체 워크플로우 요약

```
사용자 지시
    ↓
[Pre-hook] 관련 매뉴얼 자동 로드
    ↓
AI 작업 수행 (계획서/맥락노트/체크리스트 참조)
    ↓
작업 완료
    ↓
[Post-hook] 수정 파일 자동 기록
    ↓
오류 자동 검사
    ↓
셀프 체크 리마인더
    ↓
오류 발견 시 → AI 즉시 수정
    ↓
✅ 깔끔한 결과물
```

## 핵심 원칙 정리

1. **"알아서 해"는 최악의 지시다** → 계획을 먼저 세우고 검토하라
2. **매뉴얼은 자동으로 읽게 만들어라** → 후크로 강제 로드
3. **긴 매뉴얼은 목차+챕터로 분리하라** → 토큰 40~60% 절감
4. **기억은 문서에 맡겨라** → 계획서, 맥락노트, 체크리스트
5. **한 번에 하나씩 시켜라** → 작업 완료 확인 후 다음으로
6. **품질 검사는 자동화하라** → 완료 후 자동 검수
7. **역할을 나눠라** → 전문 에이전트로 팀 구성
8. **보고서를 요구하라** → "다 했습니다"로 끝내지 못하게
